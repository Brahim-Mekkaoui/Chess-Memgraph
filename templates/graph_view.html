💻 Requêtes Cypher

Copiez ces requêtes dans Memgraph Lab (port 3000).

Tous les nœuds Opening :
MATCH (o:Opening) RETURN o.eco_code, o.name ORDER BY o.eco_code LIMIT 50;
Toutes les relations :
MATCH (a)-[r:SIMILAR_TO]->(b) RETURN a.eco_code, b.eco_code, type(r) LIMIT 30;
Chemin le plus court :
MATCH p = shortestPath( (a:Opening {eco_code:'C00'}) -[:SIMILAR_TO*]- (b:Opening {eco_code:'D30'}) ) RETURN [n IN nodes(p)|n.eco_code] AS chemin;
Hubs (plus connectés) :
MATCH (o:Opening) -[r:SIMILAR_TO]-() RETURN o.name, count(r) AS deg ORDER BY deg DESC LIMIT 10;

🎓 Pourquoi une Base de Données Graphe pour les Échecs ?
🔗
Relations natives

Les relations SIMILAR_TO sont des citoyens de première classe dans Memgraph — pas des jointures SQL coûteuses. Trouver des variantes proches est une traversée de graphe, naturellement.

⚡
Performances en profondeur

Trouver un chemin entre deux ouvertures (shortestPath) est O(V+E) sur un graphe, contre O(n²) avec des auto-jointures SQL.

🧠
Modèle intuitif

Chaque ouverture = un nœud. Chaque parenté = une arête. Le schéma du graphe reflète directement la réalité des échecs — hiérarchie ECO, familles, variantes.
